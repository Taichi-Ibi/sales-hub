# Sales Hub プロダクトデザイン診断

> あるプロダクトデザイナーの「使われるプロダクトを作るための16の原則」に照らして、
> 現状の Sales Hub（Sales Intelligence）を観察・診断したもの。
> 実装の作りは丁寧だが、**「リリースストーリー」と「コア体験」が宙に浮いている**——
> これが全体を貫く問題。

作成日: 2026-06-10

---

## 🔴 最重要発見：リリースストーリーが、もう存在しない製品を指していた（原則9）

原則9「設計に迷ったらリリース投稿を考えろ。わからないものはユーザーも理解できない」。

このリポジトリには **2つの別プロダクトの痕跡** が同居していた。

|              | 旧 README が語っていた製品                                                            | 実際に動いている製品                                |
| ------------ | ------------------------------------------------------------------------------------- | --------------------------------------------------- |
| コンセプト   | 案件リレー型ワークフロー（職能横断・申し送り漏れ防止）                                | AI Sales Brain（インテリジェンス支援）              |
| 体験の主役   | ロール切替（営業/法務/PSE/PM/課長）、法務ゲート、🔔通知                               | ダッシュボード/インボックス/タスク/案件/Admin       |
| 参照ファイル | `src/lib/types.ts`, `events.ts`, `permissions.ts`, `src/routes/+page.svelte`(単一SPA) | `src/lib/intelligence/`, `src/routes/intelligence/` |
| 実在するか   | **`src/lib/types.ts` も `events.ts` も `permissions.ts` も削除済み**                  | これが現行                                          |

旧 README が説明していた法務ゲート・要員登録・課長確認・受注→PMハンドオフという
**いちばん面白いドラマは、コードからすでに消えていた**。
`法務|PSE|課長|要員|申し送り` を grep してもヒットは `intelligence/+layout.svelte` と
`admin/+page.svelte` のわずか2ファイル。リレーの本体は無い。

本当の入り口 `src/routes/+page.svelte` は、白紙に貼られた裸のリンク1本だった：

```html
<p><a href={resolve('/intelligence')}>Sales Intelligence へ</a></p>
```

原則6「認知 → 入り口 → 期待値 → 最初の最高の体験、を何度も通しで確認しろ」。
現状の玄関には期待値がゼロ。意思が見えない。

> **判定**：原則9「リリース時の様子が想像できないならコア体験がまだ存在していない」。
> 現行の Sales Intelligence を新しい人に説明するリリース投稿を書こうとした瞬間、
> 「で、この製品でユーザーは何に勝つんだっけ?」が出てこない。それが答え。

→ **対応済み**：README を現行プロダクトに合わせて全面改稿し、リリースストーリーを再定義した。

## 深掘り：危険信号トップ4

### ① AIが「あったら良いな」の博物館になっている（原則4）

`src/lib/intelligence/ai-engine.ts` のAIは6機能ある：タスク生成・サマリー・検索・
フェーズ検知・データ更新提案・振り返り。中身はテンプレート文字列の穴埋めで、どれも薄い。

原則4「『必要な気がする』機能も必要ない」「削って削って削り切ってなお削りすぎたと困るくらい削れ」。
6つあるということは、**どれが命がけのコアか決めていない**ということ。

> もし1つしか残せないなら? — コンセプト（申し送り漏れゼロ）から逆算すれば、答えは
> **「フェーズ移行/引き継ぎの瞬間に、漏れている申し送りをAIが指摘する」**1点のはず。
> 残り5つは、それを支えるか、消す。

### ② 勝ち筋を製品が測っていない（原則2）

原則2「どうすれば勝ちなのかを明確に。すべてに意思が必要」。
コンセプトは「申し送りが構造的に漏れない」。ところがダッシュボード（`src/routes/intelligence/+page.svelte`）
が見せるのは、フェーズ別件数・期限間近タスク・活動量という**どこのCRMにもある汎用KPI**。

肝心の「漏れ」を **ゼロにできているか/今いくつ漏れそうか** を可視化する画面がない。
勝ち条件を自分で測れない製品は、勝っているのか負けているのか永遠にわからない。

### ③ スタンスがUIに現れていない（原則10）

原則10「スタンスは見た目・テキストのトーン・情報設計の運び・使いやすさすべてに現れる」。

「申し送りは絶対に漏らさせない」という思想を持つなら、UIは**もっと押し付けがましく・
もっと言い切る**べき。今は汎用的で礼儀正しいダッシュボード。
Salesforce や Notion を差し置いてこれを使う「言い訳」が、画面のどこにも書かれていない。
旧 README にあった**法務ゲート（未入力だとボタンが押せない）こそが思想の体現**だった
——それを消したのは惜しい。

### ④ 入り口とオンボーディングの不在（原則6）

裸リンク玄関 → seed入りダッシュボードにいきなり着地。新規ユーザーは
「これは何で、私は最初に何をして、何に感動すればいいのか」が一切ガイドされない。
原則6の「最初の最高の体験」が設計されていない。

## 🟢 褒めるべき点

- **原則7（守破離）は優秀**：サイドバー・カンバン・インボックス・スレッド表示は王道UIを正しく模倣。
  `new SvelteSet()` 含め作法も守れている。発明に走らず先人の試行錯誤を借りている。
- **スコープは締まっている**：予測分析・チャットボット等に手を広げず、5画面に絞れている
  （原則4の精神は部分的に効いている）。
- **マスキング自動承認** という独自機構の芽がある。
  「機密を伏せたら承認ボトルネックを消す」——これは思想のかけら。育てる価値あり。
