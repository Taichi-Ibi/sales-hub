# Sales Hub — Sales Intelligence（営業インテリジェンス）モックアップ

> **一言で言うと**：営業のやり取り（Slack・メール・カレンダー・議事録・メモ）を案件に紐づけて、
> 「次にやるべきこと」と「案件を前に進める瞬間」をAIが指摘する。
> **申し送りが構造的に漏れないこと**を勝ち条件にしたツールです。

**フロントエンドのみ** のモックアップです。データは **ブラウザ（localStorage）** に保存され、
サーバー・バックエンド・外部API呼び出しはありません。AIも**外部通信のないシミュレーション（モック）**です。

```bash
bun install
bun run dev      # http://localhost:5173/intelligence
```

---

## なぜ作るのか（解きたい課題）

B2B営業では、案件が前に進むたびに情報が **Slack・メール・カレンダー・議事録** に散らばります。
担当が変わる、職能をまたぐ、時間が経つ——そのたびに「言った/言わない」「あの件どうなった」が発生し、
**申し送りが漏れる**。CRMは入力が面倒で形骸化し、結局チャットに本当の情報が埋もれる。

Sales Hub は「入力させない」立場を取ります。やり取りはすでにある。
**それを案件に集約し、次の一手と前進の合図をAIが差し出す**。これが提供したい結果です。

## このモックで体験できること

起動すると `/intelligence` に着地し、4つの画面（＋管理者用Admin）で構成されます。

| 画面 | 役割 | ここで起きること |
|------|------|------------------|
| 📊 **ダッシュボード** | その日を10秒で把握 | フェーズ別の案件進捗、24h以内の予定、72h以内の期限タスク、期間指定の振り返り |
| 📥 **インボックス** | 全イベントの時系列フィード | Slack/メール/カレンダー/議事録/メモを横断検索・スレッド表示。詳細から追記・コメント・マスキング |
| ✅ **タスク** | カンバン式タスク管理 | 未着手 / 進行中 / 完了 の3列。**AIが提案したタスクを承認/却下** |
| 💼 **案件** | 案件ごとの作業場 | 関連イベントのタイムライン、サマリー生成、**フェーズ移行のAI提案**、議事録/メモ追記 |
| ⚙️ **Admin** | 設定と監査 | 自動マスキングルール、許可メールドメイン、担当者ルール、操作ログ |

イベントは**追記保存（イベントソーシング志向）**で、案件詳細の右側に**消えないタイムライン**として残ります。

---

## コア体験の流れ（デモの一例）

1. **インボックス** を開く。未読のSlack/メールにバッジが付いている。
2. イベントをクリックすると詳細パネルが開き、**AIがタスクを提案**する
   （例：「B社 システム要件ヒアリング結果に返信する」高優先度・3日後）。
3. 提案を **承認** すると、タスクが **タスク**画面のカンバンに入る。
4. イベントを **案件** に紐づける。
5. **案件** 詳細で「サマリーを生成」すると、関連イベントから現状が要約される。
6. 議事録やメールが溜まると、AIが **フェーズ移行を提案**する
   （例：「『課題の特定』から『メリットの訴求』への移行が示唆されます」）→ 承認/却下。
7. **ダッシュボードの振り返り** で、期間内の活動量・完了タスク・フェーズ変化と改善提案を確認する。

> このモックのAIはLLMではなく**テンプレート駆動のシミュレーション**です（`src/lib/intelligence/ai-engine.ts`）。
> タスク生成・サマリー・検索・フェーズ検知・データ更新提案・振り返りを、外部通信なしで模擬します。

## ドメインモデル（要点）

- **EventLog** — `source`（slack / email / calendar / minutes / memo）、本文、`status`（pending/approved/rejected）、
  マスキング、案件への紐づけ、追記・コメント。スレッド情報（Slack `threadTs` / メール `inReplyTo`）。
- **Deal** — 名前・担当・現在フェーズ・フェーズ履歴・サマリー。
- **DealPhase（8段階）** — 商談の見極め → 課題の特定 → メリットの訴求 → 意思決定者の賛同 →
  リスクの排除 → 契約合意 → 事務処理 → 受注成約完了。
- **Task** — タイトル・案件・状態（未着手/進行中/完了）・優先度（高/中/低）・期限・出所（ai/manual）・提案フラグ。

## アーキテクチャ

```
src/
  lib/intelligence/
    types.ts          ドメイン型定義
    constants.ts      VALIDATION, STORAGE_KEYS, PHASE_LABELS, DEAL_PHASES
    store-logic.ts    純粋関数リデューサー（テスト対象）
    store.svelte.ts   Svelte 5 runes ストア（薄いラッパー、副作用）
    ai-engine.ts      AIシミュレーション（モック、外部通信なし）
    __tests__/        fast-check プロパティテスト
  lib/ui/             共通UI（Nav など）
  routes/intelligence/
    +layout.svelte    サイドバー + localStorage 初期化
    +page.svelte      ダッシュボード
    inbox/ tasks/ deals/ admin/
```

純粋関数は `store-logic.ts`、副作用（localStorage保存・スケジュール）は `store.svelte.ts` に分離。
シードデータは初回起動時に自動投入されます（冪等）。

詳細な開発ガイドは [`CLAUDE.md`](./CLAUDE.md) を参照。

## 技術スタック

SvelteKit（Svelte 5 runes）+ TypeScript + `adapter-static`（SPA、`fallback: index.html`）。
パッケージマネージャーは **bun** のみ。

```bash
bun run dev          # 開発サーバー（http://localhost:5173/intelligence）
bun run build        # 静的ビルド（build/ に出力）
bun run check        # 型チェック（0 errors が合格基準）
bun run test:prop    # プロパティテスト（fast-check）
bun run lint         # prettier + eslint
```

> モックアップのため、認証・通知・永続化・AIは実サービスに接続していません。すべて模擬です。
